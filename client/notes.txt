
CSS RESET
Problem:
Every browser (Chrome, Firefox, Safari, Edge) has its own default CSS
(e.g. body margin, button styles, font styles).

Solution:
Create reset.css to remove browser defaults so every browser starts from
the same baseline.

Common Reset Rules

*,
*::before,
*::after {
    box-sizing: border-box;
}

* {
    margin: 0;
    padding: 0;
}

Reason:
- Removes default spacing.
- Makes width calculations predictable.
- Applies to both real elements and pseudo-elements.

-----------------------------------------------------------

::before and ::after

Problem:
Sometimes we need an extra element only for decoration
(line, circle, underline, badge, background shape, etc.).

Without pseudo-elements:
Need extra HTML.

<div class="card">
    <span></span>
    Content
</div>

With pseudo-elements:
CSS creates an imaginary element.

.card::before {
    content: "";
}

Advantages
- Cleaner HTML
- Decorative elements stay inside CSS
- Widely used for UI effects

Difference

Pseudo Class
Changes the STATE of an existing element.

:hover
:focus
:first-child

Pseudo Element
Creates an imaginary element.

::before
::after
::placeholder
::selection

margin-inline: auto-
Problem:
A fixed-width container stays aligned to the left by default.

.container {
    width: 1000px;
}

Solution:
Tell the browser to automatically divide the remaining horizontal space.

.container {
    width: 1000px;
    margin-inline: auto;
}

Equivalent to:

margin-left: auto;
margin-right: auto;

Purpose:
- Horizontally centers an element.
- Modern replacement for margin-left/right.
- Uses logical properties (works with both LTR and RTL languages).


CSS TRANSITION
Problem:
CSS properties change instantly.

Gray
↓

Green

(No animation)

-----------------------------------------------------------

Solution:
Use transition.

transition DOES NOT change any property.

It only tells the browser:

"If this property changes in the future,
animate the change."

Example

a {

    transition: color 150ms ease;

}

a:hover {

    color: green;

}

Flow

Mouse Hover
↓

:hover becomes active
↓

color changes
↓

transition detects the change
↓

browser animates color over 150ms

Important

:hover changes the property.

transition animates HOW the property changes.

CENTERING A FIXED/ABSOLUTE ELEMENT
left: 50%

Moves the LEFT EDGE of the element to 50% of the parent's width.

It DOES NOT center the element.

-----------------------------------------------------------

transform: translateX(-50%)

Moves the element left by 50% of ITS OWN width.

Unlike left:50%, this percentage is calculated using the
element's width, NOT the parent's width.

Combining both:

left: 50%;
transform: translateX(-50%);

Result:

1. Left edge reaches the center.
2. Element moves back by half of its own width.
3. The CENTER of the element is now perfectly aligned with
   the center of the screen.

Remember:

left:50%      → Parent's width
translateX()  → Element's own width

WHY margin-left ON .site-content ?
The left rail uses

position: fixed;

Fixed elements are removed from the normal document flow.

Therefore the browser starts the page from x = 0,
causing the Hero to appear underneath the rail.

Solution:

.site-content {
    margin-left: var(--side-rail-width);
}

This shifts the ENTIRE website to the right by the width
of the navigation rail.

Instead of moving every section individually, we move the
parent (<main>), and all child sections automatically
follow.

-----------------------------------------------------------

On Mobile

The rail disappears.

Therefore the extra left margin is no longer needed.

@media (max-width:900px){
    .site-content{
        margin-left:0;
    }
}

CSS UNITS
===========================================================

1. px (Pixels)

Purpose:
Fixed size.

Example

width: 300px;

Meaning:
Always 300 pixels. Does not scale.

Common Uses:
- Borders
- Icons
- Small spacing
- Exact dimensions


-----------------------------------------------------------

2. % (Percentage)

Purpose:
Relative to the parent element.

Example

.parent {
    width: 1000px;
}

.child {
    width: 50%;
}

Result

Child width = 500px

If parent changes to 600px,

Child automatically becomes 300px.

Remember:

% → Parent's size


-----------------------------------------------------------

3. em

Purpose:
Relative to the current element's inherited font-size.

Example

.parent {
    font-size: 20px;
}

.child {
    font-size: 2em;
}

Result

2 × 20 = 40px

If parent font-size changes,
the child changes too.

Remember:

em → Parent's font-size (cascades)


-----------------------------------------------------------

4. rem (Root EM)

Purpose:
Relative to the root (html) font-size.

Usually

html {
    font-size: 16px;
}

Therefore

1rem = 16px

2rem = 32px

3rem = 48px

Unlike em, rem does NOT keep multiplying.

Remember:

rem → Root (html) font-size

Preferred for:
- Font sizes
- Margin
- Padding
- Gap


-----------------------------------------------------------

5. vw (Viewport Width)

Purpose:
Relative to the browser width.

Example

Browser Width = 1200px

width: 50vw;

Result

600px

Remember:

vw → Browser Width


-----------------------------------------------------------

6. vh (Viewport Height)

Purpose:
Relative to browser height.

Example

Browser Height = 900px

height: 100vh;

Result

900px

Common Uses:
- Full-screen sections


-----------------------------------------------------------

7. dvh (Dynamic Viewport Height)

Purpose:
Modern replacement for vh.

Problem:

100vh behaves inconsistently on mobile browsers because
the address bar appears/disappears.

100dvh always represents the visible viewport.

Preferred for modern websites.

Example

min-height: 100dvh;


-----------------------------------------------------------

8. fr (Fraction)

Works ONLY in CSS Grid.

Example

grid-template-columns: 1fr 2fr;

Available width = 900px

1 + 2 = 3 fractions

Result

300px

600px

Remember:

fr → Fraction of remaining space


-----------------------------------------------------------

9. auto

Purpose:
Let the browser calculate the value.

Example

margin-inline: auto;

Browser automatically calculates equal left/right margins.

Common Uses:
- Centering
- Flexible layouts

POSITION: RELATIVE-
Purpose:
Moves an element WITHOUT removing it from the normal layout.

Example

.box {

    position: relative;

    top: 20px;
    left: 30px;

}

The element is visually moved,
but its original space is still reserved.


POSITION: ABSOLUTE-
Purpose:
Removes the element from the normal document flow.

Example

.box {

    position: absolute;

    top: 20px;
    right: 30px;

}

RELATIVE + ABSOLUTE-
Most common pattern

.parent {

    position: relative;

}

.child {

    position: absolute;

    top: 0;
    right: 0;

}

The absolute element is positioned relative to
its nearest positioned ancestor.

If no positioned ancestor exists,
the browser uses the page (viewport/initial containing block).


Decorative Elements (Relative + Absolute)-
Pattern

.parent {
    position: relative;
}

.child {
    position: absolute;
}

Purpose

Decorative elements should not affect layout.

Examples

- Hero circles
- Background shapes
- Floating icons
- Badges
- Corner ribbons

INTERSECTION OBSERVER & SCROLL SPY-
Purpose:
Tracks which section of the page is currently in the viewport window
to highlight the corresponding navigation link dynamically.

Concept:
It acts as a "virtual camera" that monitors DOM elements. When
an element intersects with the specified viewing area (screen focus),
it fires a callback function to update React state.

Parameters:
- rootMargin: Expands or shrinks the boundary box of the viewport camera.
  e.g., rootMargin: "-25% 0px -55% 0px" defines a thin horizontal trigger
  strip in the upper-middle section of the viewport.
- threshold: The percentage of the target element's visibility required
  to trigger the callback (e.g., 0.1 means 10% visible).

Example Integration:
1. Initialize observer inside useEffect.
2. Watch list of section elements by ID: document.getElementById(id).
3. Update state with active element ID.
4. Clean up observer: observer.disconnect() on component unmount.

-----------------------------------------------------------

PREVENTING REDUNDANT JUMPS-
Concept:
If the user is already viewing a section, clicking its navigation link
should not trigger any scroll jump.

Implementation:
Check if the clicked link's target ID matches the active state:
if (activeSection === targetId) {
    e.preventDefault(); // Stop standard browser jump
}

MOBILE-FIRST RESPONSIVE DESIGN (BREAKPOINTS)
Purpose:
Build styles starting from mobile screens (narrowest default), and progressively add complexity for wider viewports using media queries (Tablet, Desktop).

Standard Breakpoints:
- Default: Mobile layout (< 640px)
- Tablet upgrade: @media (min-width: 640px)
- Desktop upgrade: @media (min-width: 1024px)
- Large Desktop upgrade: @media (min-width: 1440px)

Advantages:
- Incredibly clean and readable CSS files.
- Much easier to maintain than device-driven media queries (like min-width: 375px).
- Keeps site load performance high by reducing complex layout overrides.

-----------------------------------------------------------

PREVENTING MOBILE ELEMENT OVERLAPS
Problem:
Absolutely positioned sub-elements (like absolute .section-number) often collide with text on mobile screens because horizontal space is constrained.

Solution:
Stack them vertically by default (mobile-first block flow), and absolute-position them on the top-right only when screen widths expand:

.section-number {
    display: block;
    margin-bottom: 8px; /* Stack above title */
}

@media (min-width: 640px) {
    .section-number {
        position: absolute;
        top: 0;
        right: 0;
        margin-bottom: 0; /* Float to top-right on tablet */
    }
}

KEYBOARD ACCESSIBILITY (:focus-visible)
Problem:
Default browser focus outlines can look messy when clicking buttons with a mouse. If we completely disable outlines (outline: none), keyboard-only users cannot navigate the site.

Solution:
Use the CSS :focus-visible pseudo-class. It only draws an outline ring when the user interacts via a keyboard (like the Tab key), keeping normal mouse clicks clean:

a:focus-visible,
button:focus-visible {
    outline: 2px solid var(--credit-bright);
    outline-offset: 4px;
}

DNS PRECONNECT & FLASH OF INVISIBLE TEXT (FOIT) OPTIMIZATION
Problem:
1. CSS @import Blocks: Loading fonts inside CSS files using `@import` blocks style evaluation. The browser cannot render the page until the stylesheet is loaded, parsed, and fonts are fetched, causing slow page loads.
2. Flash of Invisible Text (FOIT): By default, browsers hide text while waiting for custom web fonts to download. If the connection is slow, the user sees a blank screen or missing blocks of text (FOIT), which ruins the user experience.

Solution:
1. DNS Preconnect links in index.html: Fetch the Google Fonts via optimized HTML `<link>` tags with `rel="preconnect"`. This performs DNS prefetching and early socket connection negotiations in parallel:
   <link rel="preconnect" href="https://fonts.googleapis.com" />
   <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
2. display=swap Parameter: Append `&display=swap` to the Google Fonts link query string. This tells the browser:
   "If the custom web font is not loaded yet, immediately display a system fallback font (like sans-serif) so text is instantly readable, and swap it to the custom font as soon as it downloads."
   This replaces Flash of Invisible Text (FOIT) with a safe, accessible Flash of Unstyled Text (FOUT).

Security Safeguards Implemented:
1. Helmet Middleware (server/):
Installed and mounted helmet inside app.js
to set secure HTTP response headers (preventing clickjacking, content type sniffing, HSTS bypasses)

2. NoSQL Injection Prevention (server/):
Installed and mounted express-mongo-sanitize inside app.js
to scrub incoming request payloads of MongoDB operators (keys starting with $ or .)

3.Input Sanitization & XSS Prevention (server/):
Appended .escape() sanitization inside contactValidation.js
to encode raw script tags and HTML character sequences, neutralizing stored cross-site scripting (XSS) payloads.

4.Dynamic CORS Configurations (server/):
Swapped the hardcoded CORS origin config inside app.js with a dynamic environment variable process.env.CLIENT_URL || "http://localhost:5173".

5.Secure External Links Audited (client/):
Run a full audit across all React files in client/src/ and verified that 100% of external links targeting a new tab (target="_blank") securely declare the rel="noopener noreferrer" attribute.

SEO (SEARCH ENGINE OPTIMIZATION) IMPLEMENTATION

1. robots.txt
- Purpose: Directs search engine web crawlers (like Googlebot) on which paths they are allowed to index on your site.
- Implementation: Placed in the public/ folder (copied to build root). Tells bots where your sitemap.xml is located.

2. sitemap.xml
- Purpose: Provides a structured XML list of all URLs on your website to ensure search engines can discover, crawl, and index your pages.
- Implementation: Placed in public/ public directory.

3. favicon.svg
- Purpose: Custom branding icon loaded in the browser tab.
- Implementation: Reference linked inside index.html. Adding the SVG prevents console 404 resource errors and integrates branding.
